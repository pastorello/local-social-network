## Decisions (beyond the letter of the spec)

1. **Admins always see results**, also on open polls they haven't voted in (F3.3 names only
   citizens and closed polls). Rationale: Â§3 defines admin as citizen + manage polls; deciding
   when to close one requires seeing turnout. Recorded as a parenthetical in spec F3.3.
2. **Option text â‰¤ 200 chars** (spec silent; mirrors the question cap). Blank options are
   rejected, duplicates too (case-insensitive) â€” both obvious authoring mistakes.
3. **`closes_at` must be in the future** at creation time; an instantly-closed poll is a
   misconfiguration, not a use case.
4. **State-based denials are 403 with an Italian `detail`** (voting on a closed poll, voting
   twice) â€” same pattern as M2's "report no longer editable". Invalid/foreign option ids are
   400 field errors. Double votes are also guarded by the DB constraint (`unique(user, poll)`),
   so a race collapses to the same 403.
5. **A poll is closed if `is_closed` OR `closes_at` has passed**; the API serializes the
   effective state as `is_closed`. `PATCH /:id/close/` is idempotent (closing twice is a no-op).
6. **F3.4 ordering** (open first, then closed, newest first within each group) is done with a
   `closed_rank` annotation so it holds across pagination.
7. **Vote/close respond with the full refreshed poll** so the frontend swaps one object into
   its list â€” no client-side count math.
8. **No frontend poll-detail route**: voting is inline in the list cards (a small city has few
   polls; a second page adds a hop for nothing). `GET /api/polls/:id/` exists per Â§8 anyway.
