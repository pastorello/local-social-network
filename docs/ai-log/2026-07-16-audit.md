## 1. Assumption verification detail

### A1 — Does login/registration work end-to-end? PARTLY

Files inspected: `frontend/src/views/LoginView.vue`, `SignupView.vue`, `EditProfileView.vue`, `EditPasswordView.vue`, `frontend/src/stores/user.js`, `frontend/src/App.vue`, `frontend/src/forms/user.ts`, `backend/account/api.py`, `backend/account/views.py`, `backend/account/forms.py`, `backend/account/urls.py`, `backend/backend/settings.py`.

**Login (statically sound).** `LoginView` → zod validation → `POST /api/users/login/` (`TokenObtainPairView`) → `userStore.setToken` → sets `Authorization` default header → `GET /api/users/me/` → `setUserInfo` → redirect to `/`. `App.vue` restores the header from `localStorage` on boot; `stores/user.js` calls `/api/users/refresh/` on init. The chain is complete and the endpoints exist. Runtime not exercised in this session (see Method note), but recent git history ("fix polls & users url", "showing posts is ok") is consistent with a working app.

**Registration (works, with two real problems).**
1. `account/api.py:signup` creates the user with `is_active=False` and emails an activation link. `EMAIL_BACKEND` is the **console** backend, so in dev the link only appears in the runserver terminal — a new user cannot self-activate through the UI. simplejwt refuses inactive users, so until activation, login fails.
2. **Bug:** `SignupView.vue` calls `router.push('/login')` in its success branch but never imports/creates `router` (`useRouter` is not imported). The success toast fires, then a `ReferenceError` is swallowed by the surrounding `.catch` — the user stays on the signup page.

**Activation endpoint fragility:** `account/views.py:activateemail` does `User.objects.get(id=..., email=...)` with no error handling → an invalid/expired link returns a 500 instead of a message. It also mutates state on GET.

**Profile management (broken).** `EditProfileView.vue` is a half-migrated Options-API component: it references `$refs.file.files[0]` inside `<script setup>` (undefined → throws on submit) and shadows the outer `formData` ref with a local `const formData = new FormData()`, then reads `formData.value.name` off the `FormData` object (undefined). Editing name/email/avatar cannot work as written. `EditPasswordView.vue` is wired correctly (multipart `FormData` ↔ backend `request.POST`).

### A2 — Auth implementation: JWT via simplejwt. VERIFIED

Files: `backend/backend/settings.py`, `backend/account/urls.py`, `frontend/src/stores/user.js`, `frontend/src/App.vue`, `frontend/src/views/LoginView.vue`.

- `rest_framework_simplejwt.authentication.JWTAuthentication` is the global auth class; `IsAuthenticated` is the global permission class — every endpoint is auth-required unless explicitly opened (only `signup` and the two token views are).
- Token endpoints live at `/api/users/login/` and `/api/users/refresh/` (spec §8 drafts `/api/auth/*` — see corrections).
- `SIMPLE_JWT`: access 30 days, refresh 180 days, no rotation. Both tokens persist in `localStorage`; the `Authorization` header is set in three places (`App.vue` init, login success, refresh success). There is **no axios 401 interceptor**; refresh happens only on full app reload. The 30-day access lifetime masks this today.
- **Consolidation recommendation:** keep JWT (it is the smallest change per spec §6); shorten the access lifetime and add a refresh interceptor when hardening (M1/M4), not in M0.

### A3 — State management: Pinia. VERIFIED

Files: `frontend/src/main.ts`, `frontend/src/stores/user.js`, `frontend/src/stores/toast.js`, `frontend/package.json`.

Pinia 3 registered in `main.ts`; two options-API stores. Both are plain JavaScript (untyped) and persist state by hand-mirroring into `localStorage` (`setItem('')` on logout rather than `removeItem`). No Vuex, no plugin-based persistence. The spec's §6 "verify (A3)" can be closed as "Pinia, confirmed".

### A4 — Polls/articles depth and reusability. MIXED

Files: `backend/polls/*` (models, views, api, serializers, urls, admin), `backend/posts/*`, `backend/notifications/*`, `frontend/src/views/PollsView.vue`, `PostsView.vue`, `PostView.vue`, `components/cards/QuestionCard.vue`, `components/forms/FeedForm.vue`, `components/cards/FeedItem.vue`.

**Polls: minimal and partly dead.**
- Models are the Django-tutorial pair `Question`/`Choice` with `votes = IntegerField` — **no per-user vote tracking**, so F3.2 (one final vote per citizen) and F3.3 (results gated on having voted) are unimplementable on this schema. No `created_by`, `closes_at`, or `is_closed` either.
- `polls/views.py` (`DetailView`, `ResultsView`, `vote`) renders `polls/detail.html` / `polls/results.html`; **no `.html` template exists anywhere in the backend** (`find backend -name "*.html"` → empty), so the three URL-mapped views can only raise `TemplateDoesNotExist`. They are dead code sitting under `/api/polls/<pk>/...`.
- The only live poll API is `POST /api/polls/list/` (list-via-POST, internally names its queryset `users` — copy-pasted from the user search). Frontend `PollsView` lists and searches questions; `QuestionCard` renders text + date only. **There is no voting UI at all.**
- Reusability verdict: admin registration (`polls/admin.py`) is decent; everything else is effectively a rewrite for F3 (new `Poll`/`PollOption`/`Vote` schema per spec §7).

**Posts ("articles"): the most complete feature — best reuse base for IssueReport.**
- Working loop: list (with `#hashtag` trend filter), detail with comments, create with one image attachment (`FeedForm` → multipart → `PostForm`+`AttachmentForm`), like (server-side double-like guard), comment (+notification), delete (ownership-scoped queryset — correct), report (`reported_by_users` M2M), trends.
- Direct analogue for spec §7 `IssueReport`: author FK, optional photo, denormalized counters (→ upvotes), `reported_by_users` (→ moderation). The create/serialize/list pattern transfers.
- Defects noted for the record: `is_private` is stored but **never filtered** — private posts appear in the public list and detail; counters (`posts_count`, `likes_count`, `comments_count`) are incremented without transactions and never decremented on delete; `post_create` returns `{'error': 'add somehting here later!...'}` on validation failure.

**Notifications: backend-only dead weight.**
- `grep -r "notifications" frontend/src` → **zero frontend consumers**; nothing in `AppHeader.vue` either.
- `create_notification` handles only `post_like`/`post_comment`; the three friend-request notification types and the `friendrequest_id` parameter reference a friendship feature that **does not exist anywhere in the repo** — hitting that branch would raise `UnboundLocalError` (`body` unbound). This is the closest thing to the "chat prototype" the spec expects to remove (see §2.1).

### A5 — DB engine and migrations. VERIFIED: SQLite, healthy

Files/commands: `backend/backend/settings.py`, `backend/*/migrations/`, `manage.py check` (clean), `showmigrations` (all `[X]`), `makemigrations --check --dry-run` ("No changes detected").

- Dev DB is SQLite (`db.sqlite3`, present on disk, correctly gitignored). PostgreSQL is not configured anywhere; nothing in the models needs Postgres-only features (spec §6 already rules out PostGIS; plain lat/lng floats work on both engines).
- Migration state is clean: 4 app migration chains, all applied, models in sync. `.venv` (Python 3.14) exactly matches `requirements.txt` (`pip freeze` identical, 9 pinned packages, Django 6.0.3, DRF 3.17.1 — recent and mutually consistent).
- Note: backend README says `python3 -m venv .env` but the actual venv is `.venv` at the **repo root**, and the superuser snippet omits the `python manage.py shell` invocation line.

### A6 — TypeScript coverage. REFUTED (higher than ~10%), but the gate is broken

Files/commands: every `.vue`/`.ts`/`.js` under `frontend/src`, `tsconfig.app.json`, `npm run type-check`, `npx eslint .`, `npx oxlint .`.

- **Actual coverage:** 15 of 24 SFCs use `<script setup lang="ts">`; `router`, `main.ts`, `forms/user.ts`, `composables/useFileUpload.ts`, and all 7 interface definitions are TS. Remaining JS: the 2 Pinia stores plus 9 SFC script blocks (`App`, `AppHeader`, `ActionButton`, `Toast`, `LoginView`, `SignupView`, `EditProfileView`, `EditPasswordView`, `UsersView`). The "~10%" figure in the spec is stale.
- **`strict` is already on** (inherited from `@vue/tsconfig/tsconfig.dom.json`, plus `noUncheckedIndexedAccess`) — spec N1's config goal is effectively met for typed files.
- **But `npm run type-check` fails with 15 errors** (mostly TS7016: TS files importing the untyped `.js` stores and untyped-JS SFCs; plus a real type error in `PostView.vue` and the broken `HelloWorld.spec.ts` import). Since `npm run build` runs `type-check`, **the production build currently fails.**
- JS script blocks are a blind spot: neither vue-tsc nor the current ESLint config catches the `router`-undefined bug in `SignupView.vue` or the `$refs` bug in `EditProfileView.vue` — both live in untyped `<script setup>` blocks. Converting them to `lang="ts"` surfaces such bugs at compile time.
- **Migration effort estimate:** bounded — 11 files. The 2 stores are the real work (typed state + actions, ~half day together); the 9 SFC `lang="ts"` flips are mostly mechanical (~half day), except `LoginView`/`SignupView`/`EditProfileView` which also contain the functional bugs above. Total ≈ 1–1.5 days including fixing what the compiler surfaces.

### 2.5 Code that conflicts with the spec

1. **Visitor role (spec §3) vs everything:** global `IsAuthenticated` + every meaningful route behind `requiresAuth` — an anonymous visitor can browse nothing today.
2. **Public user pages exist (spec F1.4 says none in MVP):** `UsersView` (directory + search, exposing emails) and `ProfileView`/`/profile/:id` routes.
3. **Polls schema (spec §7):** current `Question`/`Choice` has no vote attribution, no closing, no creator → F3.1–F3.3 unimplementable without the new schema.
4. **API surface (spec §8):** real prefixes are `/api/users|posts|polls|notifications/`, auth at `/api/users/login|refresh/`, list-via-POST, no pagination, ad-hoc error shapes (`{'message': ...}`, stringified form errors) vs spec's `{detail, fields}`.
5. **Posts/feed app is not in the MVP at all:** spec §4 has no posts requirement and §2 non-goals don't mention it either — yet Posts is a main nav item and the most-developed feature. Its fate (repurpose into reports vs keep vs remove) is undecided. **Open product decision — see §6.**
6. **Signup email activation is not in the spec** (F1.1 mentions only email+password with uniqueness) but is implemented and currently gates login behind a console-only email. Keep-or-drop is an open decision.
7. **README TODOs conflict with spec non-goals** (groups & invitations are a README TODO but an explicit spec non-goal).
